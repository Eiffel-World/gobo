Issues

Entity resolution, location information and SYSTEM ids.

Several problems with the architecture of the Gobo XML parser, involving the resolution of external entities, the provision of base URIs, and location information for errors, will cause me problems.

In addition, I need a means to make absolute URIs given a relative URI and a base URI. Berend's EPX_URI class will do just that.

These two issues are related, as the latter will be needed by the former.

Awaiting Eric to review EPX_URI for inclusion in Gobo, and Franck to find a solution to the first problem (I have a solution sketched out in my mind).

Numbers

A basic XSLT processor must support xs:decimal, xs:integer and xs:double. The first two require at least 18 digits of precision (but could be unlimited. The latter is the IEEE double-precision type. DOUBLE should support this, though the compilers lack routines such as is_nan, for instance.

Eric has emailed Paul Crismer re. incorporating EDA into gobo.

Exceptions

It looks like I will need to use exceptions to deal with dynamic errors. Can these be considered a violation of the contract? I don't think so.

Style

I am trying to be more consitent with my coding style. In addition to the Gobo guidelines, I name all parameters an local variables by prefixing with the indefinite article, except for booleans and plurals, plus a few notable exceptions (such as other).

In addition, I am trying to stick to a fixed order for features:

  • {NONE} -- Initialization
  • Access
  • Events
  • Measurement
  • Comparison
  • Status report
  • Status setting
  • Cursor Movement
  • Parsers
  • Creation (i.e. factory routines
  • Optimization
  • Evaluation
  • Matching
  • Element change
  • Removal
  • Resizing
  • Conversion
  • Duplication
  • Restricted (features exported to other classes)
  • Local (features exported only to this class)
  • {NONE} -- Implementation
